How we built AI-powered workflows that don't get in the way

Sam Chen - Staff Engineer Engineering

Most AI features in PM tools fall into the same trap: they generate slop that someone has to clean up. We took a different bet. Our AI doesn't produce — it triages. It reads the diff, the conversation, and the calendar, then it cuts the noise.

The first system we shipped was the auto-assigner. Given a new issue, it picks the engineer most likely to ship it well. The signal we use isn't commits/week — that just rewards busy people. It's a weighted blend of recent edits to the touched files, declared expertise, and current load.

The pipeline

Every issue passes through three stages: enrichment, scoring, and surfacing. Each one is a separate worker on a queue, so we can roll any of them back independently when they misbehave.

// pipeline/score.ts export async function scoreCandidates(issue: Issue, team: Member[]) { const signals = await gatherSignals(issue); return team .map(m => ({ m, s: scoreFor(m, signals) })) .sort((a, b) => b.s - a.s) .slice(0, 3); }
// TipThe biggest unlock wasn't the model — it was scoping the model. We give it 200 tokens of context, not 20,000. Fast, cheap, predictable.

What we got wrong

The first iteration auto-merged the AI's pick. We pulled that back in a week — engineers don't want to be assigned, they want to be recommended. The fix was tiny: change the verb. The system now suggests; a human accepts.

An AI that asks permission is a teammate. An AI that takes action is a tax.

If you've shipped AI features into a workflow tool, we'd love to compare notes. The post-mortems are more useful than the launch posts.
